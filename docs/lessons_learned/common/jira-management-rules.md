# Jira 관리 규칙

## 핵심 원칙

> **부족하면 물어보고, 설계 내용 없으면 캐물어서 작성한다.**

---

## 이슈 단위 기준

| 이슈 유형 | 기간 | 설명 |
|----------|------|------|
| **Epic** | 1개월 | 월 단위 큰 목표 |
| **Task** | 1주 | 주 단위 실행 가능한 작업 |

---

## 상태별 엄격도

| 상태 | 엄격도 | 설명 |
|------|--------|------|
| **Backlog** | 🟢 러프하게 | 아이디어 단계, 대략적인 내용만 |
| **To Do** | 🟡 상세하게 | 실행 준비 단계, 상세 내용 필수 |
| **In Progress** | 🔴 엄격하게 | 실행 중, 모든 조건 충족 필수 |

### Backlog → To Do 전환 시 (상세화 필수)

Backlog에서 To Do로 전환할 때 **엄격한 조건 적용:**

```
□ A.C. 체크박스 형식으로 작성되어 있는가?
□ 승인 기준이 명확하고 구체적인가?
□ 설계 내용이 충분한가? (없으면 캐물어서 작성)
□ 기간이 1주 이내인가?
□ Epic에 연결되어 있는가?
```

---

## 이슈 생성 규칙

### Backlog 생성 (러프하게)

- 아이디어/요청 단계
- 대략적인 Summary와 Description만으로 생성 가능
- A.C.는 간단하게 또는 생략 가능

### To Do/In Progress 생성 (엄격하게)

**필수 항목:**

| 항목 | 필수 | 내용 |
|------|------|------|
| Summary | ✅ | `[카테고리] \| 작업명` 형식 |
| Description | ✅ | 개요, 배경, 설계 내용 |
| A.C. | ✅ | 체크박스 형식 승인 기준 |
| Start date | ✅ | 시작일 |
| Due date | ✅ | 종료일 (1주 이내) |
| Epic Link | ✅ | 상위 에픽 연결 |

### 생성 시 질문 프로세스

**부족한 정보가 있으면 반드시 물어본다:**

```
1. "어떤 작업인가요?" → Summary 확정
2. "왜 필요한가요?" → 배경/목적 파악
3. "어떻게 구현하나요?" → 설계 내용 확보
4. "완료 기준이 뭔가요?" → A.C. 작성
5. "언제까지인가요?" → 기간 확정
6. "어떤 Epic에 속하나요?" → Epic 연결
```

**설계 내용이 없으면 계속 캐물어서 작성:**

```
- "구체적인 구현 방식이 어떻게 되나요?"
- "어떤 API/화면이 필요한가요?"
- "예상되는 산출물이 뭔가요?"
- "테스트는 어떻게 진행하나요?"
```

## A.C. 작성 가이드

### 형식

태스크 A.C.는 **반드시 체크박스 형식**으로 작성:

```
- [ ] 구체적인 완료 조건 1
- [ ] 구체적인 완료 조건 2
- [ ] 산출물 또는 결과 확인
```

### 예시

**Bad (모호함):**
```
- [ ] 개발 완료
- [ ] 테스트
```

**Good (명확함):**
```
- [ ] API 엔드포인트 구현 (/api/v1/events)
- [ ] 단위 테스트 작성 (커버리지 80% 이상)
- [ ] 코드 리뷰 완료
- [ ] 스테이징 배포 및 검증
```

## 이슈 완료 규칙

### 핵심 원칙

**태스크 완료 시 반드시 확인해야 할 4가지:**

1. **A.C.(Acceptance Criteria) 내용 검토** - 부족하면 보완 제안
2. **A.C. 체크박스** - 모두 DONE 처리
3. **Description 체크박스** - 모두 DONE 처리  
4. **상태 변경** - In Review 또는 Done으로 변경

---

## A.C. 내용 검토 (상태 변경 전 필수)

**In Review 상태 변경 전에 A.C. 내용을 검토하고, 부족하면 보완 제안:**

### 검토 기준

| 항목 | 확인 내용 |
|------|----------|
| 완료 기준 명확성 | 체크박스가 구체적인 완료 조건을 담고 있는가? |
| 누락 항목 | 실제 수행한 작업이 모두 반영되어 있는가? |
| 결과물 링크 | 산출물(문서, PR, 배포 등) 링크가 있는가? |

### 보완 제안 예시

```
현재 A.C.:
- [x] 기성 처리

제안:
- [x] 기성 처리
- [x] SRM 승인 완료
- [x] 결재 승인 완료 (2026-02-05)
- [x] 세금계산서 발행 확인
```

### 보완 프로세스

1. A.C. 내용 확인
2. 부족한 항목 발견 시 **사용자에게 보완 제안**
3. 사용자 승인 후 A.C. 업데이트
4. 체크박스 완료 처리
5. 상태 변경

---

## A.C. 필드 규칙

- A.C. 필드는 **반드시 체크박스 형식**으로 작성
- 태스크 완료 시 **모든 체크박스를 DONE으로 변경**
- ADF(Atlassian Document Format)의 taskItem state를 "TODO" → "DONE"으로 변경

```json
{
  "type": "taskItem",
  "attrs": {
    "localId": "...",
    "state": "DONE"  // TODO → DONE
  }
}
```

## 태스크 완료 처리 체크리스트

```
□ A.C. 필드 조회 (customfield_14516)
□ A.C. 내용 검토 → 부족하면 보완 제안
□ (사용자 승인 후) A.C. 보완 업데이트
□ A.C. 체크박스 모두 DONE 처리
□ Description 체크박스 확인
□ Description 체크박스 모두 DONE 처리
□ 상태 변경 (In Review: 5, Done: 6)
□ 보고자 멘션 댓글 (필요시)
```

## 주의사항

- sandbox 제한으로 /tmp 파일 쓰기 실패 가능 → **직접 JSON으로 API 요청**
- A.C.와 Description은 별도 필드 → **둘 다 확인 필수**
- 체크박스 상태 확인: `jq '.. | select(.localId?) | {localId, state}'`
